# Техническое задание на MVP «AI Pricing Assistant для 1С»

**Версия:** 0.1  
**Дата:** 2026-07-15  
**Назначение документа:** описать минимально жизнеспособный продукт для внедрения AI-рекомендаций по ценообразованию в 1С.  
**Рабочее название продукта:** AI Pricing Assistant для 1С.

---

## 1. Краткое описание MVP

**AI Pricing Assistant для 1С** — это модуль, который анализирует данные из 1С по товарам, продажам, ценам, остаткам и себестоимости, рассчитывает рекомендованные цены, объясняет причины рекомендации и позволяет пользователю создать черновик документа изменения цен.

Ключевой принцип MVP:

> AI не меняет цены автоматически. AI формирует рекомендацию, объясняет её и передаёт решение человеку.

---

## 2. Цель проекта

Создать MVP, который помогает коммерческому отделу, категорийным менеджерам и руководителям быстрее принимать решения по изменению цен на основе данных 1С.

Система должна:

1. Выявлять товары, по которым есть потенциал изменения цены.
2. Рассчитывать рекомендуемую цену.
3. Показывать ожидаемый эффект по марже/выручке.
4. Объяснять ключевые факторы рекомендации.
5. Сохранять рекомендацию и статус решения.
6. Позволять создать черновик документа изменения цен в 1С.

---

## 3. Бизнес-проблема

В компаниях, использующих 1С, пересмотр цен часто выполняется вручную:

- менеджеры смотрят отчёты по продажам, остаткам, закупочным ценам и марже;
- сравнивают товары с конкурентами вручную или нерегулярно;
- редко учитывают сезонность и скорость продаж;
- не всегда фиксируют причину изменения цены;
- не измеряют эффект после изменения.

В результате компания может:

- терять маржу из-за устаревших цен;
- держать завышенные цены на товарах с падающим спросом;
- поздно реагировать на рост себестоимости;
- не видеть товары, где цену можно безопасно поднять;
- тратить много времени на ручной анализ.

---

## 4. Ожидаемый бизнес-результат MVP

После внедрения MVP компания должна получить:

1. Список товаров с рекомендациями по изменению цен.
2. Понятное объяснение для каждой рекомендации.
3. Возможность быстро принять, отклонить или отправить рекомендацию на согласование.
4. Черновик документа установки цен в 1С.
5. Историю AI-рекомендаций и решений пользователей.
6. Базовую оценку экономического эффекта.

Пример целевых KPI пилота:

| KPI | Целевое значение для MVP |
|---|---:|
| Сокращение времени ручного анализа цен | на 30–50% |
| Доля рекомендаций, принятых пользователями | от 20% |
| Доля рекомендаций с объяснением факторов | 100% |
| Доля рекомендаций с сохранённым статусом решения | 100% |
| Рост валовой маржи на пилотной группе | измеряется в пилоте |

## 5. Границы MVP

### 5.1. Входит в MVP

1. Подключение к данным 1С по товарам, ценам, продажам, себестоимости и остаткам.
2. Расчёт рекомендаций по цене для выбранного товара или группы товаров.
3. Расчёт доверия/надёжности рекомендации.
4. Расчёт ожидаемого влияния на маржу.
5. Формирование объяснения рекомендации.
6. Сохранение результата в регистр/таблицу рекомендаций.
7. Интерфейс просмотра рекомендаций в 1С.
8. Действия пользователя:
   - принять;
   - отклонить;
   - отправить на согласование;
   - создать черновик документа изменения цен.
9. Логирование входных данных, выходных данных и версии модели.
10. Базовая панель контроля качества рекомендаций.

### 5.2. Не входит в MVP

1. Полностью автоматическое изменение цен без участия человека.
2. Самостоятельное проведение документов AI-системой.
3. Сложная динамическая оптимизация цен в реальном времени.
4. Автоматический парсинг цен конкурентов, если у клиента нет готового источника данных.
5. Глубокий чат-агент по всей 1С.
6. Автоматическое дообучение модели без контроля администратора.
7. Поддержка всех нестандартных доработок 1С без отдельного анализа.

## 6. Целевые пользователи

- анализировать динамику маржи.

### 6.3. Администратор 1С / IT-специалист

Задачи:

- настроить подключение;

- просматривать технические логи.

### 6.4. Data/AI-администратор

Задачи:

- контролировать качество модели;
- запускать переобучение;
- проверять версию модели;
- анализировать ошибки и обратную связь.

## 9. Принцип работы LLM в MVP

В MVP LLM может использоваться только как слой объяснения и диалога, но не как источник расчёта.

Разделение ответственности:

| Компонент | Ответственность |
|---|---|
| 1С | данные, документы, права, бизнес-процессы |
| ML/Skill-сервис | расчёт цены, доверия, эффекта, ограничений |
| LLM | объяснение результата на естественном языке, ответы на вопросы пользователя |
| Пользователь | принятие решения, согласование, проведение документов |

Запрещено:

- просить LLM «самостоятельно посчитать цену»;
- использовать LLM-ответ как единственный источник рекомендованной цены;
- давать LLM право проводить документы без подтверждения пользователя.

## 10. Основные пользовательские сценарии

### Сценарий 1. Расчёт рекомендации для одного товара

1. Пользователь открывает карточку товара или форму рекомендаций.
2. Нажимает кнопку **«Рассчитать AI-рекомендацию»**.
3. 1С собирает параметры товара.
4. 1С отправляет запрос в AI Pricing Service.
5. Сервис возвращает рекомендуемую цену, доверие, эффект и объяснение.
6. 1С показывает результат пользователю.
7. Пользователь выбирает действие:
   - принять;
   - отклонить;
   - отправить на согласование;
   - создать черновик изменения цены.

### Сценарий 2. Массовый расчёт рекомендаций по группе товаров

1. Пользователь выбирает группу номенклатуры.
2. Указывает параметры:
   - период анализа;
   - целевую стратегию;
   - минимальную надёжность;
   - максимальный процент изменения цены.
3. Нажимает **«Рассчитать рекомендации»**.
4. 1С отправляет список товаров в AI Pricing Service.
5. Сервис возвращает рекомендации по каждому товару.
6. 1С показывает таблицу рекомендаций.
7. Пользователь фильтрует рекомендации и выбирает товары для применения.

### Сценарий 3. Создание черновика документа изменения цен

1. Пользователь отмечает одну или несколько рекомендаций.
2. Нажимает **«Создать документ установки цен»**.
3. 1С создаёт документ в статусе черновика/не проведён.
4. В документ попадают:
   - товар;
   - текущая цена;
   - новая цена;
   - основание — AI-рекомендация;
   - ссылка на запись рекомендации.
5. Пользователь проверяет документ и проводит его вручную или отправляет на согласование.

### Сценарий 4. Отклонение рекомендации

1. Пользователь открывает рекомендацию.
2. Нажимает **«Отклонить»**.
3. Указывает причину:
   - не согласен с ценой;
   - есть договорные ограничения;
   - товар участвует в акции;
   - данные некорректны;
   - другое.
4. Причина сохраняется для последующего анализа и улучшения модели.

### Сценарий 5. Контроль эффекта после применения

1. Через 7/14/30 дней после изменения цены система рассчитывает фактический эффект.
2. Сравнивает прогноз и факт:
   - продажи;
   - выручку;
   - валовую прибыль;
   - маржу;
   - оборачиваемость.
3. Показывает результат:
   - рекомендация сработала;
   - эффект ниже ожидаемого;
   - недостаточно данных;
   - возникли внешние факторы.

## 11. Функциональные требования

### FR-1. Получение данных из 1С

Система должна получать по товару следующие данные:

| Поле | Обязательность | Источник |
|---|---|---|
| product_id | обязательно | справочник Номенклатура |
| product_name | обязательно | справочник Номенклатура |
| product_group | обязательно | справочник/иерархия Номенклатура |
| current_price | обязательно | регистр цен |
| cost_price | обязательно | себестоимость/закупочная цена |
| logistics_cost | желательно | расчёт/реквизит/доп. сведения |
| stock_quantity | обязательно | остатки на складах |
| sales_last_30_days | обязательно | продажи |
| sales_last_90_days | обязательно | продажи |
| sales_last_180_days | желательно | продажи |
| gross_margin_percent | обязательно | расчётное поле |
| discount_percent_avg | желательно | продажи/скидки |
| return_rate | желательно | возвраты |
| supplier_id | желательно | поставщик |
| supplier_reliability | желательно | расчётный показатель |
| competitor_price_avg | опционально | внешний источник/регистр сведений |
| seasonal_index | желательно | расчётный показатель |
| target_margin_percent | обязательно | настройка по группе |

Если часть необязательных данных отсутствует, система должна использовать fallback-значения и снижать показатель надёжности.

---

### FR-2. Расчёт рекомендации по цене

Система должна возвращать:

- текущую цену;
- рекомендованную цену;
- процент изменения;
- нижнюю и верхнюю границу рекомендуемого диапазона;
- ожидаемую маржу;
- ожидаемое изменение валовой прибыли;
- уровень риска;
- надёжность рекомендации;
- объяснение факторов;
- версию модели;
- рекомендованное действие.

---

### FR-3. Поддержка бизнес-стратегии

Пользователь должен иметь возможность выбрать стратегию расчёта:

1. **Максимизация маржи** — поднять прибыльность без критичного падения спроса.
2. **Максимизация выручки** — найти цену, которая увеличивает оборот.
3. **Распродажа остатков** — снизить цену для ускорения продаж.
4. **Сохранение конкурентности** — держать цену около рынка.
5. **Премиум-позиционирование** — поддерживать верхний ценовой сегмент.

Для MVP обязательна стратегия:

- максимизация маржи.

Остальные стратегии могут быть реализованы как настройки правил или как этап 2.

---

### FR-4. Бизнес-ограничения

Система должна применять ограничения до выдачи рекомендации пользователю.

Минимальные ограничения:

1. Новая цена не должна быть ниже себестоимости + минимальная маржа.
2. Изменение цены не должно превышать заданный процент без согласования.
3. Рекомендации с низкой надёжностью должны помечаться как требующие проверки.
4. Товары в активной акции должны либо исключаться, либо помечаться особым флагом.
5. Товары с недостаточными продажами должны получать низкую надёжность.
6. Товары с ручной блокировкой цены не должны попадать в автоматическое применение.

Настройки ограничений:

| Параметр | Значение по умолчанию |
|---|---:|
| Минимальная маржа | 20% |
| Максимальное повышение без согласования | 10% |
| Максимальное снижение без согласования | 15% |
| Минимальная надёжность для принятия | 0.70 |
| Минимальное количество продаж за 90 дней | 10 |
| Период контроля эффекта | 14 дней |

---

### FR-5. Объяснение рекомендации

Для каждой рекомендации система должна формировать объяснение.

Объяснение должно включать:

- краткий вывод;
- основные положительные факторы;
- основные негативные факторы/риски;
- ожидаемый эффект;
- ограничения;
- рекомендуемое действие.

Пример:

```text
Рекомендуется повысить цену с 179.00 до 189.90.

Причины:
- текущая цена ниже средней цены конкурентов на 6.8%;
- спрос за последние 30 дней выше среднего;
- остаток товара ограничен на 12 дней продаж;
- маржа ниже целевой на 4.2 п.п.

Риск: средний.
Цена приближается к верхней границе рынка, поэтому рекомендуется тест на 14 дней.
```

- дата расчёта;

### FR-7. Карточка рекомендации

При открытии рекомендации пользователь должен видеть:

1. Исходные данные.
2. Результат расчёта.
3. Объяснение.
4. Факторы влияния.
5. Ограничения и предупреждения.
6. Историю действий пользователя.
7. Кнопки действий.

Кнопки:

- **Принять**;
- **Отклонить**;
- **Отправить на согласование**;
- **Создать документ установки цен**;
- **Пересчитать**;
- **Показать JSON расчёта** — для администратора.

---

### FR-8. Создание документа установки цен

При принятии рекомендации система должна уметь создать черновик документа 1С.

Документ:

- тип зависит от конфигурации;
- для 1С:УТ 11 обычно используется документ установки цен номенклатуры;
- документ создаётся непроведённым;
- пользователь проводит документ вручную.

В документ должны попадать:

- номенклатура;
- вид цены;
- новая цена;
- дата установки;
- комментарий с ссылкой на AI-рекомендацию;
- ответственный пользователь.

### FR-9. Статусы рекомендаций

Рекомендация должна иметь один из статусов:

| Статус | Описание |
|---|---|
| Новая | рассчитана, ещё не рассмотрена |
| Просмотрена | пользователь открыл рекомендацию |
| Принята | пользователь согласился |
| Отклонена | пользователь отклонил |
| На согласовании | отправлена руководителю |
| Применена | создан и проведён документ изменения цен |
| Истекла | рекомендация устарела |
| Ошибка | расчёт невозможен или данные некорректны |

Система должна сохранять:

- дату и время расчёта;
- пользователя, инициировавшего расчёт;
- входные данные;
- выходные данные;
- версию модели;
- версию правил;
- статус рекомендации;
- действия пользователя;
- причину отклонения;

- файл модели в формате `.pkl`/`.joblib` или MLflow registry на более зрелом этапе;
- запуск сервиса в Docker или на отдельном сервере.

## 14. Модель расчёта цены

### 14.1. Подход для MVP

Для MVP рекомендуется гибридный подход:

1. **Бизнес-правила** — гарантируют безопасные ограничения.
2. **ML-модель** — прогнозирует допустимую/рекомендованную цену или эффект.
3. **Модуль объяснения** — формирует факторы рекомендации.
4. **Оценка надёжности** — снижает доверие при плохих или неполных данных.

### 14.2. Базовая логика

MVP может рассчитывать несколько кандидатных цен:

1. Цена по целевой марже.
2. Цена относительно конкурентов.
3. Цена с учётом продаж и остатков.
4. Цена с учётом ограничений максимального изменения.

Затем система выбирает итоговую рекомендацию через модель/правила.

Упрощённый пример:

```text
price_by_margin = cost_price / (1 - target_margin_percent)
price_by_market = competitor_price_avg * market_position_factor
price_by_stock = current_price * stock_adjustment_factor
raw_recommended_price = weighted_average(...)
final_price = apply_business_constraints(raw_recommended_price)
```

### 14.3. Что должна возвращать модель

Модель не должна возвращать только одно число. Она должна возвращать структурированный результат:

```json
{
  "recommended_price": 189.9,
  "price_min": 183.5,
  "price_max": 197.2,
  "confidence": 0.82,
  "risk_level": "medium",
  "expected_margin_percent": 32.6,
  "expected_profit_change": 1240.0,
  "factors": [
    {
      "name": "market_position",
      "impact": "positive",
      "description": "Текущая цена ниже средней цены конкурентов"
    }
  ],
  "model_version": "pricing_v1.0.0"
}
```

## 20. Обработка ошибок

Типовые ошибки:

| Код | Описание | Действие |
|---|---|---|
| DATA_MISSING | не хватает обязательных данных | показать пользователю, какие данные отсутствуют |
| PRICE_LOCKED | цена заблокирована | рекомендацию не применять |
| PROMO_ACTIVE | товар в акции | пометить как риск или исключить |
| LOW_SALES | мало продаж | снизить надёжность |
| MODEL_UNAVAILABLE | модель недоступна | повторить позже, записать ошибку |
| SERVICE_TIMEOUT | таймаут сервиса | показать ошибку, не блокировать 1С |
| INVALID_INPUT | некорректные входные данные | записать в лог, показать сообщение |

## 21. Требования к объяснимости

Для каждой рекомендации должны быть доступны:

1. Итоговая рекомендованная цена.
2. Почему цена выше/ниже текущей.
3. Какие факторы повлияли сильнее всего.
4. Какие данные отсутствовали.
5. Какие ограничения были применены.
6. Почему рекомендация считается надёжной или ненадёжной.

Нельзя показывать пользователю только фразу:

```text
AI рекомендует цену 189.90.
```

Минимально допустимый формат:

```text
AI рекомендует цену 189.90, потому что текущая цена ниже рынка, спрос растёт, а маржа ниже целевой. Надёжность 82%. Риск средний.
```

## 22. Контроль фактического эффекта

После применения рекомендации система должна уметь рассчитать факт через заданный период.

Периоды:

- 7 дней;
- 14 дней;
- 30 дней.

Показатели:

- продажи в штуках до/после;
- выручка до/после;
- валовая прибыль до/после;
- маржа до/после;
- средний чек/средняя цена;
- остаток в днях продаж;
- отклонение от прогноза.

Результат записывается в рекомендацию.

## 23. Метрики качества модели

Минимальные метрики:

| Метрика | Описание |
|---|---|
| MAE | средняя абсолютная ошибка прогноза цены/эффекта |
| MAPE | средняя процентная ошибка |
| R² | качество регрессии, если используется регрессионная модель |
| Acceptance Rate | доля принятых рекомендаций |
| Rejection Rate | доля отклонённых рекомендаций |
| Realized Uplift | фактический прирост маржи/прибыли |
| Forecast vs Actual | сравнение ожидаемого и фактического эффекта |

Для MVP важно измерять не только ML-метрики, но и бизнес-метрики.

## 24. Критерии приёмки MVP

MVP считается успешно реализованным, если:

1. Пользователь может рассчитать рекомендацию по одному товару из 1С.
2. Пользователь может рассчитать рекомендации по группе товаров.
3. Рекомендации сохраняются в 1С.
4. Для каждой рекомендации есть объяснение.
5. Пользователь может принять или отклонить рекомендацию.
6. При принятии можно создать черновик документа изменения цен.
7. AI-система не проводит документ автоматически.
8. Все вызовы AI-сервиса логируются.
9. Есть настройка минимальной надёжности и ограничений изменения цены.
10. Есть endpoint `/health` и `/model_info`.
11. Ошибка AI-сервиса не приводит к падению 1С.
12. В пилотной группе можно посчитать фактический эффект после применения.

- правила расчёта;
- требования к модели;

Результаты:

- FastAPI-сервис;

- модуль правил;
- базовая ML/эвристическая модель;
- логирование;
- тестовые данные.

### Этап 3. Разработка расширения/обработки 1С — 10–20 рабочих дней

Результаты:

- форма настроек;

Результаты:

- проверка прав;
- проверка производительности;
- сверка расчётов с бизнес-пользователями.

Результаты:

- запуск на ограниченной группе товаров;

- анализ принятых/отклонённых рекомендаций;
- расчёт фактического эффекта;
- решение о промышленном внедрении.

| Некачественные данные по себестоимости | высокая | высокое | валидация данных, предупреждения, исключение товаров |
| Нет данных конкурентов | средняя | среднее | MVP работает без них, но снижает надёжность |
| Пользователи не доверяют AI | средняя | высокое | объяснения, аудит, режим рекомендаций без автоприменения |
| Слишком много доработок 1С | средняя | высокое | начать с одной конфигурации и пилотной группы |
| Модель даёт спорные рекомендации | средняя | высокое | бизнес-ограничения, ручное подтверждение, логирование |
| Невозможно измерить эффект | средняя | среднее | заранее определить контрольную группу и период оценки |

---

## 28. Принципы безопасности и контроля

1. AI не должен иметь права самостоятельного проведения документов.
2. Все рекомендации должны быть объяснимыми.
3. Все действия должны логироваться.
4. Любая рекомендация должна иметь версию модели.
5. Пользователь должен видеть уровень риска и надёжность.
6. Рекомендации с низкой надёжностью не должны попадать в массовое применение.
7. Должен быть механизм отката через стандартные средства 1С.

## 29. Будущее развитие после MVP

После успешного MVP можно добавить:

1. Чат-помощника внутри 1С.
2. Прогноз спроса.
3. Рекомендации закупок.
4. Анализ маржинальности.
5. Автоматический сбор цен конкурентов.
6. A/B-тестирование цен.
7. Рекомендации скидок для клиентов.
8. Скоринг контрагентов.
9. Автоматическое создание задач менеджерам.
10. Полуавтономного агента по управлению ценами.

## 30. Итоговая формулировка MVP

MVP «AI Pricing Assistant для 1С» — это безопасный рекомендательный модуль, который использует данные 1С для расчёта оптимальных цен, объясняет рекомендации, сохраняет историю решений и помогает пользователю создавать черновики документов изменения цен.

Главная ценность MVP:

> Быстрее находить товары с потенциалом роста маржи и принимать ценовые решения на основе данных, не передавая AI право автоматически менять учётные данные в 1С.
